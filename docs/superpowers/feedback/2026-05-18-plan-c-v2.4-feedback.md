## Why this patch round exists

Plan C as originally scoped shipped and merged. Four real-API verification rounds confirmed every functional bug from the original review was fixed (arxiv 4.0 migration, FTS5 sanitisation, renderer fallback, tracer status, cache reuse < 500ms, multi-paper citations ≥ 2 distinct papers). Then a **browser-level manual test** by the project owner surfaced four operational gaps and **two coupled design re-prioritisations**:

1. User attaches a paper (via Add button or library browse), but the *next* `paper_qa` turn returns "No references are enabled" — frontend `session_id` never reaches the backend. **(Critical)**
2. The trace panel sits empty for 60-90 seconds during paper_search, then all rows appear at once. **(Critical)**
3. The trace panel renders only headlines — no LLM `reason`, no `query`, no `result_summary`. The "tool selection logic" demo is opaque. **(Critical)**
4. No UI to view session references, toggle, remove, browse the library — and no UI to add papers from a paper_search result. Endpoints exist, surfaces don't. **(Important)**
5. Semantic Scholar should be the primary external search; arXiv reserved for raw-source download with PDF fallback for non-arXiv papers. **(Design change)**
6. **Reverts v2.3 "default-add top 1-2" to suggest-only.** Research naturally surfaces many tangential candidates; auto-downloading them wastes time, disk, and arXiv rate budget for sources the user wouldn't have picked. Agent becomes **read-only** — three tools (`search_library`, `search_semantic_scholar`, `find_related_papers`) and shortlists 3-5 candidates per turn. User is the sole ingestion trigger via clicking Add on a `SearchResultList` card. **(Design change)**

The SRS v2.4 entry captures items 4-6 as design intent. Items 1-3 are operational bugs that block the demo flow regardless. The five tasks below combine all six findings into five executable units.

## Task v2.4-1 — Frontend session_id roundtrip

### Goal

After the user clicks Add on a candidate card (or attaches a paper from the Library Browser), the *next* `paper_qa` turn in the same browser chat must see that paper as an enabled reference. Same root cause whether the attach comes from the v2.3 agent auto-add (now removed) or the v2.4 user-driven Add click.

### Root cause

[`frontend/src/hooks/useChatStream.ts:42`](../../../frontend/src/hooks/useChatStream.ts#L42):

```ts
await streamChat({ session_id: null, user_message: userMessage, history }, ...)
```

The backend [`chat.py::_ensure_session`](../../../backend/src/paperhub/api/chat.py) treats `null` as "create a new chat_sessions row". Every turn → new session → user-attached papers belong to backend session N, next turn queries backend session N+1. Same problem affects `POST /papers` and `POST /papers/from-library` from the Add buttons — they need the **current backend** `session_id`, which means the frontend must learn and persist it on the first SSE event.

### Backend changes

**File:** `backend/src/paperhub/models/events.py`

Add a new top-of-stream event that carries the resolved backend session ID. Either:
- (a) Extend `RoutingDecisionEvent` with an optional `session_id: int | None` field, OR
- (b) Add a new `SessionEvent(type="session", session_id: int, run_id: int)` emitted as the very first SSE event before `tool_step` / `routing_decision`.

Prefer **(b)** — cleaner separation, doesn't change `RoutingDecision` semantics. Frontend learns the session_id immediately on connection rather than after router classification.

```python
class SessionEvent(BaseModel):
    type: Literal["session"] = "session"
    run_id: int
    session_id: int
```

Add to `SseEvent` union and to `sse_format` helper if it's used.

**File:** `backend/src/paperhub/api/chat.py`

After `_ensure_session` resolves `session_id` and `_new_run` resolves `run_id` (around current line 134-135), emit the new event as the very first yield:

```python
session_id = await _ensure_session(conn, req.session_id)
run_id = await _new_run(conn, session_id)
sess_evt = SessionEvent(run_id=run_id, session_id=session_id)
yield {"event": sess_evt.type, "data": sess_evt.model_dump_json(exclude={"type"})}
```

## Task v2.4-2 — Stream `tool_step` during paper_search

### Goal

Trace panel rows appear in the browser as the Research Agent executes each tool call, not all at once at the end.

### Root cause

[`backend/src/paperhub/api/chat.py:182-189`](../../../backend/src/paperhub/api/chat.py#L182-L189):

```python
elif intent == "paper_search":
    pipeline = PaperPipeline(...)
    final_content = await paper_search(state, ...)   # ← synchronous, runs ~60-90s
```

Single `await`, no `tool_step` SSE events emitted until after `paper_search` returns.

### Solution

Mirror the `paper_qa_stream` pattern: convert `paper_search` to an async generator yielding either a tool-call record or the final message.

**File:** `backend/src/paperhub/agents/research.py`

Define (or reuse if already present in the FinalOnlyMessage pattern):

```python
@dataclass
class ToolStepYield:
    """Yielded by paper_search after each tracer.step closes so the chat
    endpoint can forward as a tool_step SSE event in real time."""
    record: dict[str, Any]   # the just-persisted tool_calls row (cols match _drain_tool_calls_since)
```

Change `paper_search` signature:

```python
async def paper_search(
    state: AgentState, *, adapter, tracer, model, conn, pipeline,
) -> AsyncIterator[ToolStepYield | FinalOnlyMessage]:
    ...
```

After each `tracer.step()` context manager closes (i.e., after each `paper_search:plan` / `paper_search:search_library` / `paper_search:search_arxiv` / `paper_search:add_paper_to_session`), the just-inserted `tool_calls` row needs to be read back and yielded.

Cleanest implementation: pass the existing `_drain_tool_calls_since` helper down, or refactor it into a shared util in `tracing/tracer.py`. After each step closes, drain rows newer than `last_emitted_step`, yield each as `ToolStepYield(record=...)`.

At the very end of the loop (when the LLM returns a final message with no tool_calls), yield `FinalOnlyMessage(content=final_text)`.

**File:** `backend/src/paperhub/api/chat.py`

Replace the `paper_search` branch:

```python
elif intent == "paper_search":
    pipeline = PaperPipeline(conn, papers_cache_dir=..., chroma=get_chroma(request, settings))
    final_content = ""
    async for item in paper_search(
        state, adapter=adapter, tracer=tracer,
        model=settings.paper_qa_model, conn=conn, pipeline=pipeline,
    ):
        if isinstance(item, FinalOnlyMessage):
            final_content = item.content
        else:  # ToolStepYield
            yield {"event": "tool_step",
                   "data": json.dumps({"record": item.record}, separators=(',', ':'))}
            last_emitted_step = item.record["step_index"]
```

### Tests

**`tests/test_chat_sse.py::test_paper_search_streams_tool_step_events_incrementally`**:

Set router mock for `paper_search`. Set `litellm.acompletion` side_effect to a sequence of fake responses driving the tool loop (4-5 steps). Use an `httpx.AsyncClient` stream consumer + `time.monotonic()` to record arrival timestamps of each `tool_step` event. Assert:

- At least 3 `tool_step` events arrive
- The time between consecutive `tool_step` events is non-zero (events arrive incrementally, not all buffered at end)

A simpler assertion if timing-based tests are flaky: hook into the SSE stream and verify that *between* receiving `tool_step` event N and event N+1, the test can perform an arbitrary action (e.g., increment a counter) — proves the events are not all delivered in one chunk.

### Don't break

- The existing paper_search test (`test_research_paper_search.py`) calls `paper_search` directly and asserts on the return value. Since the function is now an async generator, the test must `async for item in paper_search(...)` and collect items. The final string is now `item.content` from the last `FinalOnlyMessage`.

## Task v2.4-3 — TraceInline expand-on-click

### Goal

Click any row in the trace panel → reveal `args_redacted_json` (with the LLM's `reason` prominent) and `result_summary_json`.

### File

`frontend/src/components/chat/TraceInline.tsx`

### Implementation sketch

Make each `<li>` itself a controlled disclosure. Track an open-set:

```tsx
const [openSteps, setOpenSteps] = useState<Set<string>>(new Set());
const toggleStep = (key: string) => setOpenSteps(prev => {
  const next = new Set(prev);
  next.has(key) ? next.delete(key) : next.add(key);
  return next;
});
```

Per-row markup (replacing the current `<li>`):

```tsx
{trace.map((r) => {
  const key = `${r.branch}-${r.step_index}`;
  const isOpen = openSteps.has(key);
  return (
    <li key={key} data-status={r.status} className={rowClasses(r.status)}>
      <button
        type="button"
        onClick={() => toggleStep(key)}
        aria-expanded={isOpen}
        className="w-full text-left flex items-center gap-1"
      >
        {isOpen ? <ChevronDown className="h-3 w-3" /> : <ChevronRight className="h-3 w-3" />}
        [{r.branch || "main"}#{r.step_index}] {r.agent} · {r.tool}{" "}
        ({r.model ?? "-"}) {r.latency_ms}ms {r.status}
        {r.error && ` — ${r.error}`}
      </button>
      {isOpen && (
        <div className="mt-1 ml-4 space-y-1 text-xs">
          <TraceArgs args={r.args_redacted_json} />
          <TraceResult result={r.result_summary_json} />
        </div>
      )}
    </li>
  );
})}
```

Two small render helpers:

- `TraceArgs({args})` — if `args.reason` exists, render it first in italics with label "Why:". Then render known-shape rows: `query`, `paper_id`, `arxiv_id`, `mode`, `max_results`. Anything else falls into a final `<pre>` JSON block.
- `TraceResult({result})` — if `result.summary` exists (the agent's loop wraps everything in `summary`), unpack it first. Render known-shape rows: `count`, `title`, `cache_hit`, `papers_id`, `paper_content_id`. Anything else falls into a `<pre>` JSON block.

Both helpers handle `null` / missing gracefully (render nothing).

**Add** `GET /papers?session_id=N` (list session's references):

```python
@router.get("", response_model=list[ReferenceItem])
async def list_session_references(session_id: int = Query(...)) -> list[ReferenceItem]:
    """List papers attached to a session, joined to paper_content for display fields."""
    async with get_conn() as conn:
        rows = await conn.execute_fetchall(
            "SELECT p.id, p.paper_content_id, p.enabled, p.added_at, "
            "       pc.arxiv_id, pc.title, pc.year, pc.kind "
            "FROM papers p JOIN paper_content pc ON pc.id = p.paper_content_id "
            "WHERE p.session_id = ? ORDER BY p.added_at DESC",
            (session_id,),
        )
    return [ReferenceItem(...) for r in rows]
```

`ReferenceItem` Pydantic model: `papers_id`, `paper_content_id`, `enabled: bool`, `added_at: str`, `arxiv_id: str | None`, `title: str`, `year: int | None`, `kind: str`.

**Extend** `POST /papers` to accept prefix-discriminated `paper_id` (this is the Add-button entry point from SearchResultList; legacy `arxiv_id` field stays for backwards-compat with existing scripts):

```python
class IngestRequest(BaseModel):
    session_id: int
    paper_id: str | None = None       # NEW (preferred): "arxiv:<id>" | "ss:<paperId>" | "library:<paper_content_id>"
    arxiv_id: str | None = None       # legacy alias for paper_id="arxiv:<arxiv_id>"

    @model_validator(mode="after")
    def _exactly_one(self):
        if not (self.paper_id or self.arxiv_id):
            raise ValueError("provide paper_id or arxiv_id")
        if self.paper_id and self.arxiv_id:
            raise ValueError("provide only one of paper_id / arxiv_id")
        return self
```

The endpoint normalises `arxiv_id` → `paper_id="arxiv:{arxiv_id}"` and dispatches through `add_paper_to_session_dispatch` (Task v2.4-5 refactors that dispatcher to take `paper_id`). On `NoIngestibleSourceError` (only possible when `paper_id` starts with `ss:`), return **HTTP 422** with body `{"detail":"no_ingestible_source","title":"...","paper_id":"..."}`.

## Task v2.4-5 — Semantic Scholar primary; PDF ingestion fallback

### Goal

Match SRS v2.4 design intent: (a) Semantic Scholar as primary external search, arXiv reserved for raw-source download with PDF fallback for non-arXiv papers; (b) **shortlist-with-finalize agent** — `add_paper_to_session` is removed from the LLM-visible palette, but the agent can mark up to 2 most-confident candidates with `finalize: true`; chat endpoint auto-attaches finalized picks (server cap), user manually adds the rest via SearchResultList cards or Library Browser; (c) new `search_results` SSE event carrying the enriched candidate list (with `auto_added`, `papers_id`, `error` fields populated post-finalize).

### Architectural shape

```

│  paper_search turn                                                   │
│                                                                      │
│  Agent (read-only loop, NO write tool):                              │
│    palette = {search_library, search_semantic_scholar,               │
```

```python
        self.title = title


async def add_paper_to_session_dispatch(
    paper_id: str, *,
    pipeline: PaperPipeline, conn: aiosqlite.Connection, session_id: int,

    """Invoked ONLY by POST /papers (Task v2.4-4) — never by the agent loop."""
    if paper_id.startswith("library:"):
        return await _attach_library(paper_id.removeprefix("library:"), conn=conn, session_id=session_id)
```

Implementation sketch:

```python
import re, json
from dataclasses import dataclass

CANDIDATES_BLOCK_RE = re.compile(r"```json:candidates\s*\n(.*?)\n```", re.DOTALL)

@dataclass
class SearchResultsYield:
    candidates: list[SearchCandidate]

# Inside paper_search, after the LLM's final no-tool-calls response:
def _extract_candidates(final_text: str, recent_results: dict[str, dict]) -> tuple[str, list[SearchCandidate]]:
    """Strip the json:candidates block from final_text, return the cleaned text
    + the parsed candidates enriched with metadata from `recent_results` (a cache
    of the most recent search_library / search_semantic_scholar / find_related_papers
    tool results, keyed by paper_id)."""
    m = CANDIDATES_BLOCK_RE.search(final_text)
    if not m:
        # Tolerant: if the agent forgot the block, yield no candidates rather than crash.
        return final_text, []
    raw_picks = json.loads(m.group(1))
    cleaned_text = CANDIDATES_BLOCK_RE.sub("", final_text).strip()
    candidates: list[SearchCandidate] = []
    for pick in raw_picks:
        meta = recent_results.get(pick["paper_id"])
        if meta is None:
            continue  # picked something we didn't return — ignore defensively
        candidates.append(SearchCandidate(
            paper_id=pick["paper_id"],
            reason=pick["reason"],
            title=meta["title"],
            authors=meta.get("authors", []),
            year=meta.get("year"),
            abstract=meta.get("abstract"),
            arxiv_id=meta.get("arxiv_id"),
            has_open_pdf=bool(meta.get("open_access_pdf_url")),
            already_in_session=pick["paper_id"].startswith("library:") is False and False,  # see below
        ))
    return cleaned_text, candidates
```

Note: `already_in_session` is computed at the chat.py layer after lookup — the agent doesn't know which library papers are already attached because `search_library` filters them out. Set the flag from the chat layer after fetching `paper_id`s that match attached `papers`.
